# Обновлённая позиция и рекомендация

## Краткая формулировка

Правильная интерпретация предыдущих пакетов такова:

- **архитектурное направление верное**;
- **масштаб немедленного внедрения избыточен**;
- **проекту сейчас нужен не новый framework, а эволюция существующего ontology-first pipeline**.

## Что следует принять как принципы

### 1. Separation of concerns

Нужно сохранить чёткое различие между:

- semantic/content layer;
- lexical / localization layer;
- realization layer.

Это уже соответствует духу текущей архитектуры и не требует революции.

### 2. Intention не должна менять истину

Интенция допустима как способ:

- выбрать формулировку;
- сменить фокус;
- подчеркнуть правило / исключение / наблюдение;
- адаптировать текст к дидактической задаче.

Но она не должна менять semantic core.

### 3. Morphology — сервис, а не центр архитектуры

Морфологический движок должен стать полезным инфраструктурным сервисом для рендера, а не отдельной “новой платформой”.

### 4. Реализация должна быть инкрементальной

Каждое улучшение должно:

- интегрироваться в уже существующие пайплайны;
- иметь локальную и измеримую пользу;
- не требовать полной architectural freeze фазы.

## Что не следует делать сейчас

### 1. Не вводить новый формальный AST ради самого AST

В проекте уже есть достаточно структурированное semantic ядро. На текущем этапе разумнее улучшать пригодность существующих структур для рендера, чем создавать ещё один слой представления.

### 2. Не выносить отдельный intention framework до появления реальной боли

Сейчас intention уже неявно закодирована в inventory шаблонов и модулей. Этого достаточно для ближайших итераций.

### 3. Не строить общий multilingual text framework авансом

Следует сначала закрыть реальные проблемы в RU vertical slice и только потом решать, какие abstraction boundaries действительно оправданы.

## Что реально нужно сейчас

### Направление 1. `decline()`

Нужен компактный rule-based morphology service, покрывающий самые частые кейсы склонения и согласования для рендера.

Это даст немедленную пользу для:

- theory references;
- task templates;
- rule summaries;
- UI text fragments.

### Направление 2. Rule summary generation

Нужно продолжить движение в сторону Phase B1: генерировать тексты правил из уже структурированных данных, не строя пока общую framework-конструкцию.

### Направление 3. Расширение morph directives в template engine

Нужно добавить те morph-директивы, которые действительно требуются существующим шаблонам, а не проектировать полный grammar DSL заранее.

## Что имеет смысл отложить

Следующие вещи можно считать **future options**, а не текущим backlog:

- явная formalization intention layer;
- отдельный generalized semantic AST;
- полноценный construction family registry как самостоятельная подсистема;
- discourse planning между предложениями;
- LLM-driven verbalization поверх semantic expressions.

## Переформулированная целевая картинка

Правильная цель на текущем этапе — не “semantic-to-text platform”, а:

> **аккуратное усиление существующего ontology-first rendering pipeline**

через небольшие, проверяемые и уже прикладные улучшения.

## Итоговая рекомендация

Рекомендуется принять оба пакета как:

- reference architecture;
- vocabulary для обсуждения;
- источник принципов и guardrails.

Но в execution перевести их в более узкий план:

1. `decline()`
2. rule summary generation
3. targeted morph directives
4. инвентаризация уже существующих template families

И только если после этого останется системная боль, возвращаться к вопросу о формализации более высоких слоёв.
