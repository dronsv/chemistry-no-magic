# Пакет документов по semantic-to-text / morphology / ontology verbalization

Этот пакет собран по итогам обсуждения идеи перехода от хранения готовых строк и локальных морфологических форм к более общей архитектуре:

- **каноническое семантическое выражение**;
- **отдельный слой intention / communicative framing**;
- **ontology-linked lexicon**;
- **construction / realization layer**;
- **surface rendering** для разных языков.

## Состав пакета

### 01_overview_neutral_summary.md
Нейтральный обзор результатов разговора: что именно обсуждалось, к каким выводам пришли, где есть согласие, а где остаются открытые решения.

### 02_target_architecture.md
Предлагаемая целевая архитектура: уровни модели, разграничение ответственности между онтологией, лексиконом, intention и renderer, а также принципиальные ограничения.

### 03_existing_analogues.md
Обзор наиболее близких существующих направлений и систем: GF, OntoLex-Lemon, ontology verbalization, classic NLG pipeline, surface realization.

### 04_project_complexity_and_scope.md
Оценка сложности проекта: что является инженерной задачей, что уже похоже на research-grade, где MVP реалистичен, а где начинается опасный рост scope.

### 05_recommendations_and_next_steps.md
Практические рекомендации: что лучше делать сначала, какие слои фиксировать отдельно, какие артефакты ввести в проекте и чего не делать на раннем этапе.

## Назначение пакета

Пакет предназначен как **обзорно-архитектурная база для обсуждения**, а не как финальная спецификация для немедленной реализации.

Документы сознательно выдержаны в нейтральном стиле:

- без жёсткой привязки к одной-единственной реализации;
- без преждевременной фиксации всех enum / schema;
- с разделением между уже согласованными принципами и вопросами, требующими отдельного ADR / spec.

## Ключевая мысль пакета

Обсуждаемая задача — это уже не просто «морфология» и не просто «локализация строк».

По сути речь идёт о построении **semantic-to-text слоя поверх химической онтологии**, где:

1. онтология хранит смысл;
2. лексикон связывает смысл с языком;
3. intention определяет, зачем и как подаётся факт;
4. construction / realization layer превращает это в грамматически корректный текст.

## Статус

Рекомендуется использовать пакет как основу для:

- дальнейшего архитектурного обсуждения;
- выделения отдельного ADR / RFC по semantic expression layer;
- выделения отдельного ADR / RFC по intention layer;
- определения MVP-границ.
